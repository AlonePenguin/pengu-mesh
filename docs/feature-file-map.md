# Feature File Map

This file maps the current capability families to the code, docs, and proof
surfaces that own them. It is the answer to “where does this feature live?” and
“where should the next change land?”

## Status legend

- `shipped`: live, verified, and part of the intended product surface
- `partial`: live but still missing meaningful execution-path coverage
- `foundation`: code exists in-tree with tests or helpers, but it is not yet a
  shipped CLI, MCP, or HTTP contract surface
- `deferred`: intentionally not shipped yet
- `placeholder`: named location exists, but no real implementation lives there yet
- `unsupported-platform`: honest unsupported path exists for non-target platforms
- `docs-only`: requirement exists only in docs or planning material

## Change checklist

When a capability changes:

1. update `docs/current-status.md`
2. update this file if the owning paths or status changed
3. update `docs/implementation-backlog.md` or `docs/milestone-plan.md` if the roadmap changed
4. decide lease coverage explicitly
5. update CLI, MCP, HTTP, shared types, and failure contracts together
6. add or extend the relevant release proof
7. store durable proof in the repo when the result matters beyond the active run
